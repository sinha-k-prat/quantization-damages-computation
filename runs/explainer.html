What does compressing an AI actually break?

An illustrated walk-through of a small experiment — with every idea shown on a real example.

To run on phones and cheaply in data-centers, AI models are quantized: every number inside them is rounded to a coarse setting. Everyone measures how much accuracy drops. Almost no one asks which specific abilities break. We built a tiny model we fully understand, quantized it, and watched — one skill at a time.

1 · The machine we built

A miniature language model — the same design as modern LLMs (Qwen/Llama family), just tiny enough to dissect completely.

Each of the 6 layers has 7 weight matrices we can quantize (the attention Q, K, V, O and the MLP's gate, up, down) → 42 total. Think of each matrix as a panel of dials the model turned during training.

2 · What "quantization" actually does — on a real matrix

Here is one real row of weights from our trained model (matrix L1.v). Quantizing to 2 bits means: instead of letting each number be anything, we pick just 4 allowed values (a "codebook") and snap every weight to its nearest one.

Original weights (full precision — each can be anything):
0.0070.0220.009−0.036−0.027−0.049−0.011−0.0040.017−0.007
2-bit codebook — only these 4 values are allowed:  −0.055   −0.020   0.006   0.037
After snapping each weight to the nearest allowed value:
0.0060.0370.006−0.020−0.020−0.055−0.0200.0060.0060.006
Layman version: imagine a volume knob that could turn to any position. Quantizing replaces it with a knob that only has 4 click-stops. Close enough for most jobs — but you can no longer make fine adjustments.

Crank it harder — 1 bit = only 2 allowed values (−0.081, 0.075) — and the same row becomes almost a coin-flip of two numbers:

0.0750.0750.075−0.081−0.081−0.081−0.081−0.0810.075−0.081

This "how coarse can a matrix's dials get before it breaks" is the whole story — and later it becomes our measuring tool.

3 · The task — and the six skills hidden in it

We feed the model a mixed list of objects and numbers and ask a question. A real example:

75coinballcard695262

Question: "2nd element after first card"  →  Answer: 52

(Find card — it's at position 4 — then count 2 further to position 6, which holds 52.)

The questions are built to exercise six distinct sub-skills, so we can test each one separately:

The clever bit: the model is never told that "ball" is round. It has to learn that hidden property — exactly like an LLM silently knowing facts about words. That makes "semantic" a genuine recall-from-memory skill, not something readable from the question.

4 · What is "pos N"? — the model showing its work

We train the model to write a scratchpad — its step-by-step working — before answering. Here is the real scratchpad for the example above. Read it like a child's long-division working:

relative   anchor first card → pos 4   "the first card sits at position 4"
index   2 after → pos 6 = 52   "2 further along is position 6, which holds 52"
read   A 52   "answer: 52"
So pos 4 is just the model writing down a position as a number. "pos N" = "the thing I'm looking for is at slot N." It's the moment the model commits the result of a hard search to paper. This turns out to be the single hardest — and most fragile — step in the whole task.

Why is writing pos 4 hard? To produce it, the model must (1) know we mean the object "card", (2) scan the list, (3) find the first one, (4) work out where it is. Everything after that number is written is easy arithmetic on a symbol it can now just read back.

6 · The headline finding: computation breaks, memory doesn't

Measuring the 2-bit penalty for each skill separately:

Plain version: squeezing the model to 2 bits barely touches what it remembers (facts, categories) but blunts what it works out (counting, comparing, locating). This matches the folk experience that quantized chatbots stay fluent and factual but get worse at math and multi-step reasoning.

7 · Why? — one simple rule

Quantization is a brittle extrapolator. Inside the range it was trained on, coarse dials are fine (they even help, like rounding away noise). Push beyond that range and the coarse dials can't reach the in-between settings needed — and computation, which produces ever-larger numbers on bigger inputs, is exactly what runs off the edge.

This one rule explains everything above: memory skills have a small, fixed menu of answers → safe to compress. Computation skills produce quantities that grow with the input → the coarse dials can't keep up once you leave the training range.

8 · A bonus: turning the damage into a microscope

If crushing a matrix only hurts computation, we can run it backwards as a diagnostic: go to each of the 42 matrices, crush only that one to 1 bit, and see what breaks. Matrices doing delicate computation will scream; matrices doing simple lookups won't notice.

9 · Honest limits

  • This is a tiny 4.3M model on a synthetic task, one random seed, one compression level. The next step is repeating it on a real small language model (e.g. Pythia-160M) with a standard quantizer.
  • We tested a tempting stronger claim — that quantization specifically breaks "recall-and-then-use" — and a careful control experiment disproved it. We report that, because the honest version (the brittle-extrapolator rule) is what survived.
  • The effects are small in absolute terms because the model has fully learned the task; the signal is in the pattern (which skills, which matrices, which tokens), not the raw size.
